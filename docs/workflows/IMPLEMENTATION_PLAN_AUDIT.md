# Implementation Plan Audit

An implementation plan can look polished and still be bad for an agent.

For smaller models, the most common failure is a plan that is too broad. The model starts a phase, fills context with files and tool output, loses the original objective, and either loops or invents the next step.

An implementation-plan audit asks:

> Can the target model actually execute this plan under the stated context budget?

## When to audit a plan

Audit a plan before giving it to a coding agent when:

- the project is large
- the task spans multiple subsystems
- the target model has limited context
- the plan was generated by another model or tool
- the plan has only a few broad phases
- the task is a port, migration, refactor, or long-running implementation

## Core audit dimensions

Grade each dimension A-F:

| Dimension | What to check |
|---|---|
| Phase granularity | Are phases small enough for the target model and context length? |
| Atomicity | Can each phase be executed without major inference leaps? |
| Dependency order | Are prerequisites handled before dependent work? |
| Context fit | Does each phase fit the `targeted_context_length`? |
| Validation | Does each phase define concrete pass/fail checks? |
| Task memory | Does the plan write progress, decisions, artifacts, and next prompts? |
| Handoff quality | Can a new session resume from the phase state? |
| Risk control | Are Git, destructive actions, and side effects handled safely? |
| Test strategy | Are tests planned before and after implementation work? |
| Small-model executability | Could a smaller executor model carry out the orders quickly and safely? |

## What a good phase contains

Each phase should include:

- purpose
- inputs
- likely files/directories
- explicit tasks
- expected outputs
- validation commands or checks
- stop condition
- phase debrief requirements
- do-not-carry-forward notes

## Good vs weak phases

Weak:

```text
Phase 2: Port the UI to Linux.
```

Better:

```markdown
## Phase 2: Establish Linux UI startup path

Goal: Make the app launch the main UI window on Linux.

Likely files:
- `src/platform/*`
- `src/main/*`
- `package.json` or build config

Tasks:
1. Identify Windows/macOS-only startup assumptions.
2. Replace only the minimum platform-specific path needed for Linux launch.
3. Add or update one smoke test if the repo supports it.

Validation:
- Run the narrowest launch/build command available.

Stop condition:
- UI launches on Linux, or blocker is documented with exact file/error evidence.

Debrief:
- Completed
- Evidence
- Blockers
- Carry forward
- Do not carry forward
```

## Strong planner, local executor

A strong model can create or audit the plan. A smaller local model can execute it when phases are explicit enough.

The plan should reduce how much reasoning the executor has to do. The executor should be able to read the phase, inspect a small file set, make a bounded change, validate, and debrief.

## Common plan problems

- only 3 phases for a large project
- no stop conditions
- no validation per phase
- no task-state files
- no phase debrief
- too many files per phase
- no test strategy
- no Git safety
- broad “polish” phases that invite scope creep
- no handling for blockers or failed validation
